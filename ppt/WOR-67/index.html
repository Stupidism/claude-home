What is OpenAI Agents SDK?

OpenAI Agents SDK (formerly Swarm, also referred to as Symphony) is an open-source framework for orchestrating multiple AI agents that coordinate to solve complex tasks.


It provides a small, composable set of primitives — keeping the framework easy to learn while being powerful enough for production-grade multi-agent pipelines.

Design Goals

  • Minimal abstractions — few concepts, easy to learn
  • Provider-agnostic — works with OpenAI & 100+ other LLMs
  • Production-ready — tracing, guardrails, human-in-the-loop

Open Source

Apache-2.0 license · Python 3.9+ · Active development by OpenAI

Primitive 1 — Agents

An Agent is an LLM paired with:

  • Instructions — system prompt defining its role and behavior
  • Tools — functions the agent can invoke (web search, code exec, APIs…)
  • Guardrails — safety constraints applied before/after generation
  • Handoffs — pointers to other agents it can delegate to

Agents are composable — each is a discrete unit that can be combined into complex pipelines while staying independently testable.

from agents import Agent, Runner

triage = Agent(
  name="Triage",
  instructions="""
    Route the user's request
    to the right specialist.
  """,
  handoffs=[coding_agent, qa_agent]
)

result = Runner.run_sync(
  triage,
  "Fix the login bug in auth.py"
)
print(result.final_output)

Primitive 2 — Handoffs

Handoffs let one agent delegate to a specialist agent, creating an orchestration hierarchy.


How it works

  5. Result returned up the chain

Primitive 3 — Tools

Function Tools

Any Python function decorated with @function_tool. The SDK auto-generates the JSON schema.

@function_tool
def search_web(query: str) -> str:
    """Search the web."""
    return do_search(query)

MCP Tools

Connect to any Model Context Protocol server — file system, databases, APIs — with zero boilerplate.

async with MCPServerStdio(
  "npx",
  ["-y", "@modelcontextprotocol/server-filesystem", "."]
) as mcp:
  agent = Agent(mcp_servers=[mcp])

Hosted Tools

OpenAI-managed tools running in the cloud: Web Search, File Search, Code Interpreter.

agent = Agent(
  tools=[
    WebSearchTool(),
    CodeInterpreterTool()
  ]
)

Primitive 4 — Guardrails

Guardrails are safety checks that run in parallel with the agent — they can halt the run before an unsafe output reaches the user.


Input Guardrails

Validate user input before the agent processes it. Block jailbreaks, off-topic requests, or sensitive data leaks.


Output Guardrails

Check agent output before it is returned. Detect hallucinations, PII, or policy violations.

from agents import (
  Agent, GuardrailFunctionOutput,
  InputGuardrail, RunContextWrapper,
  input_guardrail
)

@input_guardrail
async def no_competitor_mentions(
  ctx: RunContextWrapper,
  agent: Agent,
  input: str
) -> GuardrailFunctionOutput:
  if "competitor" in input.lower():
    return GuardrailFunctionOutput(
      output_info="blocked",
      tripwire_triggered=True
    )
  return GuardrailFunctionOutput(
    output_info="ok",
    tripwire_triggered=False
  )

agent = Agent(
  input_guardrails=[no_competitor_mentions]
)

Primitives 5–8: Sessions · Tracing · Human-in-Loop · Realtime

💬 Sessions

Automatic multi-turn conversation state. The SDK manages message history so agents remember prior turns without manual plumbing.

🔍 Tracing

Every agent run is automatically traced. View call trees, token usage, and tool invocations in the OpenAI dashboard — no extra code needed.

👤 Human-in-the-Loop

Pause execution at any point to collect human approval or input. Resumable — the agent picks up exactly where it left off.

🎙️ Realtime Agents

Voice-enabled agents powered by gpt-realtime-1.5. All SDK features (tools, guardrails, handoffs) work in voice mode.

Motivation: Why Build This?

The Problem

  • Frontend engineering has high volume, repetitive work — many small tickets, all requiring the same cycle: read → code → test → PR
  • Each ticket takes 30–90 min of human focus even when the logic is straightforward
  • Context switching between tickets is expensive

The Insight from OpenAI Agents SDK

If an agent can be given instructions + tools + autonomy, it can handle the full software engineering loop:


  1. Read the ticket requirements
  2. Explore the codebase
  3. Write + validate the code
  4. Create the PR
  5. Respond to review feedback

The poller runs concurrently — up to N agents in parallel, each in its own git worktree

poll-linear.mts — The Brain

A TypeScript script that runs permanently on my Mac as a daemon.


Poll loop (every 30s)

  1. Fetch tickets from Linear API by state
  2. Find eligible tickets (assigned to me, in Todo or In Progress)
  3. Skip blocked tickets (missing AC, dependencies)
  4. Spawn Claude Code agents for eligible tickets
  5. Monitor running agents, handle exits

Smart features

  • Concurrent agents (configurable max)
  • Rate-limit detection & sleep-in-place
  • Auto-resume after poller restart
  • Feedback mode (rework tickets)
// Simplified core loop
while (true) {
  // Rate-limit pause (sleep in-place)
  if (rateLimitPausedUntil) {
    const ms = rateLimitPausedUntil - Date.now();
    await sleep(ms);          // blocks here
    rateLimitPausedUntil = null;
  }

  await poll();

  // For each eligible ticket:
  if (!runningAgents.has(ticket.id)) {
    await moveToInProgress(ticket);
    spawnAgent(ticket, 'fresh');
  }

  await sleep(POLL_INTERVAL_MS); // 30s
}

run-ticket.sh — The Worker

Each ticket gets its own isolated environment:


1. Git Worktree

A fresh checkout in ~/worktrees/feat/TICKET-slug/ — completely isolated from main and from other running agents.


2. Claude Code Agent

Runs claude --dangerously-skip-permissions with WORKFLOW.md injected as the system prompt. The agent reads the ticket, explores the codebase, writes code, and creates a PR.

# 1. Create isolated worktree
git worktree add \
  "$WORKTREE_PATH" \
  -b "feat/$TICKET_ID-$SLUG" \
  origin/master

# 2. Symlink node_modules (fast setup)
ln -s "$REPO_ROOT/node_modules" \
  "$WORKTREE_PATH/node_modules"

# 3. Inject WORKFLOW.md + ticket context
PROMPT=$(envsubst < ~/.claude/WORKFLOW.md)

# 4. Launch Claude Code
claude \
  --dangerously-skip-permissions \
  --print "$PROMPT"

Agent output is streamed to logs/symphony-$TICKET_ID.log for debugging.

WORKFLOW.md — The Agent's Playbook

The agent follows a structured workflow defined in ~/.claude/WORKFLOW.md:


  1. Read & Plan — fetch ticket from Linear, identify affected files, create workpad
  2. Setup Worktree — already done by runner
  3. Implement — write code, run type checks per phase
  4. Validate — lint, unit tests, app-specific proof
  5. Rebase + PR — rebase onto master, create PR with symphony label
  6. Submit for Review — post evidence to workpad, move ticket to Human Review

Skills System

Each step is a Markdown skill file in ~/.claude/skills/. The agent reads the relevant skill before each phase.


Skills are reusable across projects and repos — a create-pr skill works the same for workstream-mono as for claude-home.

Linear Workpad

Each ticket gets a persistent workpad comment — the agent's scratchpad for plans, AC checkboxes, and progress notes. Survives poller restarts.

claude-home — The Config Repo

claude-home (Stupidism/claude-home) is my personal home directory as a git repo — only the files I care about are tracked.

    CLAUDE.md      ← global agent instructions
    WORKFLOW.md    ← agent playbook template
    skills/        ← reusable skill files
    memory/        ← persistent cross-session memory

Why version-control ~/.claude?

  • Skills improve over time → track the history
  • Reproduce the setup on any machine
  • Each skill improvement is a PR — review it like code

Team-friendly design

Fork the repo, add your own boards/*.json for your Linear team, and the system works for any repo. No Workstream-specific hardcoding.

What works well

  • Small, well-defined tickets with clear AC
  • Frontend features following existing patterns
  • Bug fixes with a clear reproduction path
  • Chores: dependency updates, config changes

What still needs humans

  • Ambiguous or under-specified tickets
  • Cross-repo changes requiring coordination
  • Novel architecture decisions
  • Debugging flaky Cypress E2E tests

Key Learnings

1. Ticket quality is the bottleneck

A ticket with clear Acceptance Criteria and a reproduction case gets solved autonomously. A vague ticket gets reworked 3 times. Garbage in → garbage out.

2. Skills are your prompt engineering

Every skill improvement directly improves all future tickets. Treat ~/.claude/skills/ like production code — review, test, iterate.

3. Worktrees are the key to parallelism

Git worktrees give each agent a fully isolated environment with zero overhead. Without them, concurrency would require cloning the whole repo each time.

4. Human-in-loop is a feature, not a compromise

The PR review step catches the 20% of cases where the agent went in the wrong direction. Don't try to remove it — it's what makes the system trustworthy.

5. Sleep-in-place for rate limits

Don't return early and re-poll — sleep in-place until the rate limit window expires. This preserves the agent's session state and context.

6. The poller is the orchestrator

Like OpenAI's Triage Agent, poll-linear.mts decides what runs, when, and in what order. Good orchestration = good agent performance.

Conclusion

OpenAI Agents SDK taught me…

  • Composable, focused agents beat monolithic prompts
  • Guardrails and human-in-loop are first-class primitives
  • Tracing and observability are essential for production agents
  • The orchestrator pattern scales to any domain

My practice showed me…

  • Agents can handle the full software engineering loop
  • Ticket quality matters more than agent capability
  • Version-controlling your prompts is as important as your code
  • Human review is a feature, not a crutch

The future of software engineering isn't replacing developers — it's amplifying their output by delegating the repetitive work to agents while keeping humans in control of the decisions that matter.
